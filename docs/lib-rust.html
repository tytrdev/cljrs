Rust interop

cljrs is just a Rust crate (cljrs) and an Env you populate with Builtins. Any Rust crate can be exposed as a cljrs namespace by writing a thin shim — a pub fn install(env: &Env) that registers a few function pointers under string names. Three shipped crates use this pattern: cljrs-physics, cljrs-ml, and the wasm-side js_bridge/ui_bridge in cljrs-wasm. The cljrs-bindgen tool can generate one of these shims for you from a TOML manifest.

Core building blocks

  • cljrs::value::Value — the dynamic value enum: Nil, Bool, Int, Float, Str, Vector, List, Map, Set, Builtin, Fn, Opaque, etc.
  • cljrs::value::Builtin::new_static(name, fn(&[Value]) -> Result<Value>) — wrap a fn pointer as a callable cljrs value.
  • cljrs::env::Env — namespace registry. env.set_current_ns("cljrs.foo"), then env.define_global("bar", value) binds cljrs.foo/bar.
  • cljrs::error::{Error, Result} — fallible Rust returns surface to cljrs as evaluator errors.

Opaque handles

Native Rust values that cljrs shouldn't introspect (a Tensor, a RigidBody, a JsValue) round-trip through Value::Opaque: 

Value::Opaque {
    tag:   Arc<str>,                              // "ml/tensor"
    inner: Arc<dyn Any + Send + Sync>,             // your handle
}

Conventions in the standard crates:

  • Tag is "<ns>/<type>" — e.g. "ml/tensor", "physics.d2/world".
  • Inner is wrapped in Arc so cljrs can clone the value cheaply.
  • Mutability lives inside the inner type (a RefCell or Mutex) — cljrs sees an immutable handle and mutators are !-suffixed builtins.
  • Handles that don't impl Send + Sync (e.g. JsValue) get a newtype with an unsafe impl Send + Sync; soundness rests on the target being single-threaded.

The install pattern, by example

cljrs-physics is the model — read crates/cljrs-physics/src/lib.rs for the full text. Skeleton: 

pub fn install(env: &Env) {
    let prev = env.current_ns();
    env.set_current_ns("cljrs.physics.d2");
    bind(env, "world",          world_fn);
    bind(env, "add-body!",      add_body_fn);
    bind(env, "step!",          step_fn);
    bind(env, "translation",    translation_fn);
    // ...
    env.set_current_ns(prev.as_ref());
}

fn bind(env: &Env, name: &'static str, f: fn(&[Value]) -> Result<Value>) {
    env.define_global(name, Value::Builtin(Builtin::new_static(name, f)));
}

fn world_fn(args: &[Value]) -> Result<Value> {
    let opts = arg_map(args, 0, "world")?;
    // ...build a Rapier World...
    Ok(Value::Opaque {
        tag:   Arc::from("physics.d2/world"),
        inner: Arc::new(Mutex::new(world)),
    })
}

Three things to copy into a new crate: the install shape (save+restore current_ns), the bind helper, and a small set of arg_* coercion helpers (see cljrs-ml/src/lib.rs for arg_tensor, arg_f32, arg_usize).

Adding a new crate to the workspace

  1. cargo new --lib crates/cljrs-mything. Edit Cargo.toml to add cljrs = { path = "../.." } and whatever upstream crate you're wrapping. Re-export under [lib] crate-type = ["lib"].
  2. Write pub fn install(env: &Env) following the physics template. Put each fn *_fn(args) next to its bind line; arg coercion helpers at the bottom.
  3. Wire it into the wasm bundle if browser users should see it: in crates/cljrs-wasm/src/lib.rs, add cljrs_mything::install(&env); next to the existing installs (there are three call sites — eval_source, the Repl::new, and the reset path).
  4. Also wire it into native binaries that should see it (src/bin/demo.rs, etc.).
  5. Document the surface area: add a lib-mything.html following the existing template, and announce its functions in Coverage.

cljrs-bindgen — manifest-driven shim generator

crates/cljrs-bindgen generates an entire install file from a TOML manifest. Useful when the Rust crate is mostly stateless free functions — you describe the surface area in TOML, the bindgen emits the Rust shim, you compile. 

[crate]
name    = "rand"
version = "0.8"
ns      = "cljrs.rand"
imports = ["rand::Rng", "rand::rngs::ThreadRng as Rng"]

[[fn]]
name     = "random"
clj-name = "rand"
args     = []
returns  = "f64"

[[fn]]
name     = "shuffle"
clj-name = "shuffle"
args     = ["vec<i64>"]
returns  = "vec<i64>"

Type whitelist (v1): i64, f64, bool, string, opaque:Tag, vec<i64>, vec<f64>. Anything else is rejected at manifest-load time. Methods on opaques use [[method]] with an on = "Tag" discriminator. See crates/cljrs-bindgen/src/manifest.rs for the full schema.

For complex APIs (callbacks, async, mutable state mid-flight, anything outside the whitelist) write the shim by hand against the physics template. The bindgen aims to remove boilerplate for the easy 80%, not to be a substitute for understanding the pattern.

Errors & conventions

  • Return Err(Error::Type(..)) for argument-type mismatch, Error::Eval(..) for runtime failure, Error::Arity{expected,got} for wrong arg count.
  • Mutator builtins end in ! (Clojure convention). Predicates end in ?.
  • Constructors take a single map of options when they have more than two args — gives every call a self-documenting site.
  • Use imbl::Vector / imbl::HashMap directly when building return values; cljrs's persistent collections are imbl under the hood.

Examples in the wild

  • Physics — Rapier 2D/3D bound as cljrs.physics.d2 / .d3.
  • ML — autograd tensors as cljrs.ml.
  • JS — wasm-only DOM/event/fetch/storage as cljrs.js.
  • UI — Preact-backed mount/hydrate as cljrs.ui.

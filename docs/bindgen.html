cljrs-bindgen — manifest-driven Rust interop

Phase 1 of cljrs's Rust interop is hand-written: each foreign crate gets its own shim crate (cljrs-physics, cljrs-ml, cljrs-music) that exposes a pub fn install(env: &Env) registering builtins. Phase 2 is cljrs-bindgen: a small CLI that reads a TOML manifest and emits the same shape of crate automatically, so adding a new library is "describe the surface area, run the generator, drop it in the workspace."

Type whitelist (v1)

i64 f64 bool string vec<i64> vec<f64> opaque:Tag

Anything else (maps, custom structs, references, generics) needs a hand-written shim. The bindgen exits non-zero with a clear error if a manifest references a type outside the whitelist.

opaque:Tag wraps any Send + Sync + 'static Rust value behind a Value::Opaque. The same tag is used both for the cljrs-side type identity and (by default) the in-scope Rust type name — bring the concrete type into scope under that name via the manifest's imports field.

Manifest format

[crate]
name    = "rand"
version = "0.8"
ns      = "cljrs.rand"
# Optional `use` lines emitted at the top of the generated lib.
imports = [
  "rand::rngs::StdRng as Rng",
  "rand::SeedableRng as _SeedableRng",
  "rand::Rng as _RngTrait",
]
# Optional. Default = crate name. Set to "" to write each fn's call path
# explicitly (handy when mixing free fns with associated fns).
fn-prefix = ""

# Free functions.
[[fn]]
name     = "rand::random::<f64>"
clj-name = "random"
args     = []
returns  = "f64"

# Associated function returning an opaque handle.
[[fn]]
name     = "Rng::seed_from_u64"
clj-name = "seeded-rng"
args     = ["i64"]
returns  = "opaque:Rng"
# `call` overrides the default `name(arg0, arg1, ...)`. Substitutions
# `{0}` `{1}` ... refer to the marshalled args. Useful for casts or
# wrapper shapes (Range, Option) the foreign API requires.
call     = "seed_from_u64({0} as u64)"

# Methods on an opaque handle. Receiver is always arg 0 from cljrs's POV.
[[method]]
on       = "Rng"
name     = "gen_range"
clj-name = "gen-range"
args     = ["i64", "i64"]
returns  = "i64"
call     = "gen_range({0}..{1})"   # rand 0.8 expects a Range

Running the bindgen

cargo run -p cljrs-bindgen -- \
    crates/cljrs-bindgen/examples/rand.toml \
    --out crates/cljrs-rand/

Generates crates/cljrs-rand/Cargo.toml and crates/cljrs-rand/src/lib.rs. The CLJRS_PATH env var controls the relative path to the cljrs crate written into the generated Cargo.toml (defaults to ../.., which is correct when emitting under crates/).

Wiring it up

  1. Add the new crate to the workspace's members: 
    [workspace]
members = [".", ..., "crates/cljrs-rand"]
  2. Add a path-dep on the new crate from whichever binary owns the REPL / runtime (e.g. cljrs-rand = { path = "crates/cljrs-rand" }).
  3. Call cljrs_rand::install(&env) after constructing the Env, before user code runs.
  4. From cljrs: 
    (def rng (rand/seeded-rng 42))
(rand/gen-range rng 0 100)  ;=> 17

What the codegen emits

One Rust shim fn per cljrs builtin, sharing a small set of inline helpers (arg_i64, arg_opaque, opaque, ...). Example for the gen-range method above: 

fn method_gen_range(args: &[Value]) -> Result<Value> {
    let recv = arg_opaque::<Rng>(args, 0, "Rng", "gen-range")?;
    let a1 = arg_i64(args, 1, "gen-range")?;
    let a2 = arg_i64(args, 2, "gen-range")?;
    let r = { let mut __recv = (&*recv).clone(); __recv.gen_range(a1..a2) };
    Ok(Value::Int((r) as i64))
}
Methods clone the handle out of the Arc before dispatch — fine for cheap-to-clone handles like StdRng, but a workaround for cljrs-bindgen's lack of unique ownership tracking. Phase 3 will introduce Mutex-wrapped handles for foreign types whose &mut self APIs don't tolerate cloning.

End-to-end walkthrough: binding a new crate

  1. Pick the surface. For each function/method you want exposed, check that every arg + return is in the whitelist. If not, either: (a) wrap it in a thin Rust newtype that exposes a whitelist-friendly API, or (b) hand-write that one builtin and let bindgen handle the rest.
  2. Write the manifest. Start with fn-prefix = "" if the crate's API spans modules; that lets each fn name be a fully qualified path.
  3. For opaque handles: bring the concrete Rust type into scope via imports (e.g. "rand::rngs::StdRng as Rng"). The opaque tag (Rng) is what every opaque:Rng mention in the manifest lines up with.
  4. For traits providing methods on the handle (rand's Rng trait, serde's Serializer, etc.), import them under a non-clashing name ("rand::Rng as _RngTrait") so their methods resolve.
  5. Run cargo run -p cljrs-bindgen -- <manifest> --out crates/cljrs-<name>.
  6. Add to workspace members; cargo build -p cljrs-<name>; iterate on the manifest until clean.
  7. Call cljrs_<name>::install(&env) from the host.

Limits & deferred work

  • No map / struct marshalling yet — the runtime supports it (cljrs-physics uses keyword-keyed maps for body config), but the manifest can't express the field schema. Phase 3.
  • No Result unwrapping — foreign fns returning Result<T, E> must be wrapped by hand.
  • No version-pinning across foreign-crate releases. Bumping the bound crate's version in the manifest doesn't catch source breakage; you find out at cargo build.
  • Opaque handles are Arc<T>; methods needing unique &mut self work only if T: Clone.

Benchmarks

Numerical micro-benchmarks comparing cljrs (MLIR-JIT native path) against other Clojure-family runtimes. Please poke holes in this. Everything below is reproducible — methodology, source, runtime versions, and known biases are all explicit.

What each benchmark measures

  • fib(35) — naive recursive Fibonacci. Measures function-call overhead, integer arithmetic, branching. Pathologically dense call graph (no memoization).
  • loop_sum — a loop/recur summing 100 million integers. Measures tight-loop codegen and tail-call handling. This is where JIT compilers earn their keep.
  • cond_chain — a 10-arm cond that classifies each of 50 million values. Measures branch-prediction-friendly dispatch and conditional codegen.

Methodology

Every run uses the same source code for each runtime, the same number of iterations, and is invoked via a shell wrapper at bench/run.sh. To reproduce: 

git clone <repo> && cd cljrs
cargo build --release --features mlir --bin bench
cd bench && ./run.sh

The kernels are 1:1 ports of each other. For fib:

;; bench/fib.clj              — tree-walker
(defn fib [n] (if (< n 2) n (+ (fib (- n 1)) (fib (- n 2)))))

;; bench/fib_native.clj       — cljrs MLIR-JIT
(defn-native fib ^i64 [^i64 n]
  (if (< n 2) n (+ (fib (- n 1)) (fib (- n 2)))))

Known biases (please verify)

  • JVM startup: Clojure-on-JVM numbers exclude JVM startup. We measure the inner loop only via (System/nanoTime). If you care about startup-inclusive numbers, JVM is much worse (~1.5s cold).
  • JVM warmup: one warmup iteration is done before the timed run. A longer warmup would favor JVM slightly; we matched what most blog-post benchmarks do.
  • Babashka: uses GraalVM native-image — great for startup, poor for hot loops (no inline caches). It's fast at what it's for (shell scripting), but this isn't it.
  • jank: a C++-backed Clojure dialect. We use its time macro (millisecond granularity). Numbers below ~10ms have higher relative noise. Its JIT is also newer than Clojure's.
  • Output side-channels: we verified no printing or I/O happens inside any timed loop; all kernels return a scalar that's printed after the timer stops.
  • One machine: everything below is from a single Apple M3 Max. Results on x86_64 may differ (LLVM backend is the same, but JVM HotSpot is tuned differently).
  • Micro, not macro: these are fundamental-op benchmarks. Real programs have memory allocation, polymorphic dispatch, and persistent-collection ops that aren't exercised here. A whole-program benchmark (e.g., a small game loop) is TODO.

Raw source

Benchmark kernels live in bench/*.clj in the repo. The driver script bench/run.sh invokes each runtime with the same per-benchmark iteration count and prints a table. All of it is ~200 lines total — small enough that "you audited what?" has an honest answer.

GPU benchmarks

Elementwise dst[i] = sin(x[i]) + cos(x[i] * 2), f32, Apple M3 (integrated GPU, Metal backend). All timings are steady-state median-of-10 after warmup; readback time included. Lower is better.

N cljrs-gpu cljrs-cpu numpy pytorch-cpu pytorch-mps
100 k 1.45 ms 0.47 ms 0.33 ms 0.13 ms 0.43 ms
1 M 3.60 ms 4.37 ms 2.89 ms 0.74 ms 0.97 ms
10 M 16.2 ms 34.7 ms 27.0 ms 16.2 ms 4.43 ms
100 M 142 ms 347 ms 266 ms 256 ms 42.7 ms

cljrs-gpu = wgpu / Metal, kernel emitted from a cljrs defn-gpu form. cljrs-cpu = single-threaded plain-Rust reference. numpy = 2.4.4 on macOS Accelerate. pytorch-cpu / pytorch-mps = 2.11.0 on CPU and Apple MPS.

What this says

  • At small N (≤100k) the GPU loses: launch + readback overhead swamps the compute. Use CPU.
  • At 10–100M, cljrs-gpu beats pytorch-cpu and numpy (1.6–1.9× vs numpy at 100M), and loses to pytorch-mps by ~3×. PyTorch on MPS uses Apple's hand-tuned Metal Performance Shaders kernels; our wgpu path goes through the generic Metal backend without those.
  • Single-thread Rust (cljrs-cpu) lands 2× slower than numpy, as expected — numpy is vectorized + multithreaded via Accelerate.
  • The point isn't that cljrs-gpu beats everything — it's that we ship Clojure source that compiles to a WGSL kernel, which runs on any wgpu backend (Metal, Vulkan, DX12) and also in browsers via WebGPU. Same source, same output, four targets.

The kernel

Cljrs source → WGSL → dispatched by wgpu:

;; Grid-stride loop so a fixed dispatch covers any N — WebGPU caps
;; workgroups at 65535/dim, so we can't linearly scale dispatch.
@compute @workgroup_size(256)
fn main(@builtin(global_invocation_id) gid: vec3<u32>,
        @builtin(num_workgroups) nwg: vec3<u32>) {
    let stride = nwg.x * 256u;
    var i = gid.x;
    loop {
        if (i >= arrayLength(&src)) { break; }
        let v = src[i];
        dst[i] = sin(v) + cos(v * 2.0);
        i = i + stride;
    }
}

Caveats + known biases

  • Readback tax. cljrs-gpu includes GPU→CPU readback every call. numpy / pytorch-cpu are already on CPU. pytorch-mps pays synchronization cost but keeps tensors on GPU by default — if you'd chain multiple ops, pytorch would look even better. Our numbers are "one op with full round-trip."
  • Single kernel. Elementwise sin+cos. Doesn't exercise matmul, reductions, or stencils where the GPU gap widens. More kernels coming.
  • Apple Accelerate. numpy uses Apple's vectorized BLAS; on Linux x86 it'd use OpenBLAS or MKL. Numbers may differ.
  • CUDA not tested here — this box has no NVIDIA GPU. Planned for a benchmark rig with one.
  • WebGPU not tested here — same kernel should run in the browser via the GPU web demo, but timing a browser kernel precisely needs instrumentation we don't have yet.
